## R16 GET /gacha/pay/result

- 记录编号：`R16`
- 优先级：`最高级`
- 测试类型：`单接口-多阶段 ramp-up`
- 接口：`抽卡结果`
- 并发策略：`200 > 400 > 600`
- 启动速率：`10`
- 持续时间：`13m = 3(200) + 5(400) + 5(600)，逐渐升高`

### 关键数据

- 总请求数：`22192`
- 业务成功数：`22192`
- 失败数：`0`
- 平均 RT：`11068.66 ms`
- P95：`21000 ms`
- 成功 QPS：`28`
- Locust RPS：`32.6`
- RO QPS：`688`
- CCU 峰值：`14`
- CPU 峰值：44%

### 现象

1. 13 分钟压测期间，系统平均每秒处理约 `32` 个请求，接口层面没有直接报错。
2. 小程序出现严重卡顿，进入后白屏，加载时间需要 `15s`。
3. 对 TDSQL-C 已造成明显压力，CPU 和 CCU 飙升明显。
4. 随着 users 持续增加，QPS 基本趋于平稳，没有明显继续上涨。

### 建议研发关注

- `抽卡结果` 对应接口已经不是一般性能波动，而是明确的瓶颈点，优先级应高于本文其余问题。
- 建议优先排查：
  - `GET /gacha/pay/result`
  - 抽卡结果查询相关 SQL / 缓存 / 状态聚合逻辑
  - 高并发下数据库连接、热点数据争用、重复查询放大问题
- 该问题的特点是“错误率不高，但体验极差，同时数据库压力已经明显异常”，属于需要研发优先处理的问题。

## 给研发的统一结论

1. `R16`：`抽卡结果（GET /gacha/pay/result）` 为当前最高优先级问题。表现为白屏、加载 15 秒、数据库 CPU/CCU 飙升、继续加压时 QPS 不再明显上涨，已经是明确瓶颈。
2. `R07`：`查看商品详情` 在逐级升压下出现高 RT 和明显前端白屏/卡顿，属于“错误率不高但体验明显退化”的问题。
3. `R40`：`卡册图鉴页` 对应接口吞吐很低、耗时极高，是当前最明显的单接口性能瓶颈之一。
4. `R41`：完整业务链路在压测下虽然未大面积报错，但已经出现整体卡顿和支付结果超时，需要从链路视角排查而不是只看单接口状态码。
